# Plan: Contract conformance test (offline — required)

## Context
Prove that observe-js serializes exactly what `observe-contract@v0.1.2` pins — field-for-field equality against `golden-record.json`, `fixtures/service-start.json`, and `levels.json` — with no live backend. This `vitest` suite gates the build and is the reference harness `observe-swift`/`observe-dart` will copy, so it must stay clean and self-contained.

It flagged one **environment-confirmed defect** (the crypto stub) and two **capture-ordering clarifications**. All three are folded into the design decisions and tasks below.

## Key design decisions (read before implementing)

The golden fixtures contain **fixed** values (timestamps, instance id, trace/span ids) but the SDK injects non-deterministic ones (`Date.now()` for `timeUnixNano`/`observedTimeUnixNano`, a random UUID for `service.instance.id`, random ids for spans). The strategy is to **control the inputs deterministically and normalize nothing on the output** — every field is produced naturally by the real SDK code path and compared with a strict deep-equal:

- **Clock:** `vi.useFakeTimers()` + `vi.setSystemTime(1718632800000)`. `nowNanoString()` in `src/core/sdk.ts` computes `BigInt(Date.now()) * 1_000_000n`, so `1718632800000` ms → `"1718632800000000000"` ns, matching the fixtures' `timeUnixNano` and `observedTimeUnixNano` exactly. (Review confirmed fake timers are compatible: the batcher's `setInterval` is harmlessly faked, `flush()` is promise-based not timer-coupled, and the exporter's `AbortSignal.timeout(5000)` never fires because the mocked `fetch` resolves immediately.)
- **Instance id (CRITICAL — environment-confirmed defect in review):** stub the **whole `crypto` global** with `vi.stubGlobal('crypto', { randomUUID: () => '0b9d7a3e-5f2c-4c1a-9e7d-3a6b8c1f2e4d' })`. ⚠️ Do **not** use `vi.spyOn(globalThis.crypto, 'randomUUID')` — `globalThis.crypto` is `undefined` inside the vitest `node` VM on this machine (Node 18.15.0, probe-confirmed), so `vi.spyOn` throws in `beforeEach`; and even if it didn't, `newInstanceId()` in `src/core/resource.ts` would fail its `typeof globalThis.crypto?.randomUUID === 'function'` guard and fall through to the `Math.random` UUID fallback, producing an id that never equals the fixture. `vi.stubGlobal` *defines* the global when absent (the same pattern `exporter.test.ts` already uses for `fetch`), which both makes `globalThis.crypto` exist and routes `resource.ts` down the `randomUUID` branch — making the `service.instance.id` (`0b9d7a3e-…`) in the `resource` block of **both** fixtures byte-identical. The `globalThis.crypto?.` optional chaining and Math.random fallback in `resource.ts` are load-bearing on this runtime — do not simplify them away. Cleared by the `vi.unstubAllGlobals()` in `afterEach`.
- **Trace/span ids:** the golden record's `traceId`/`spanId`/`flags` are the canonical W3C traceparent example. Reconstruct the context with `extract(objectCarrier({ traceparent: '00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01' }))` and run the `log()` call inside `runWithContext(ctx, …)`. `log()` stamps `record.traceId`/`spanId`/`flags` from the active context (sdk.ts lines 162–167) — exact ids, no normalization.
- **Wire capture:** mock global `fetch` to capture each POST body string and return `new Response(null, { status: 200 })` (the exporter reads only `.status` and accepts 200/204; review used this form successfully — use it consistently, not the bare `{ status: 200 }` object). The default exporter (`src/core/exporter.ts`) calls `JSON.stringify(encodeLogs(resource, records))` and POSTs it — capturing the body gives the **exact on-the-wire payload** through the full real pipeline (`buildResource` → batcher → `encodeLogs` → `JSON.stringify`). `JSON.parse` it and `expect(...).toEqual(fixture)`.
- **Context manager registration:** import the **Node entry** (`src/node/index.js`) for its side-effecting `setContextManager` registration (review confirmed `src/node/context.ts` calls it at module load), otherwise the no-op manager leaves `getActiveContext()` undefined and the trace ids never stamp. Hence the `.node.test.ts` suffix (matches `context.node.test.ts`/`winston.node.test.ts`).
- **Single-record envelopes + marker-before-record ordering (capture-indexing fix from review):** each fixture is a one-record `ExportLogsServiceRequest`. The SDK is `init`'d once (module singleton — re-init is a no-op) and `init` enqueues the `service.start` marker. To keep each captured request a single-record envelope, **use one ordered `it` block** that: (1) `init` → `await flush()` → captured body `[0]` is the `service.start` request; (2) `runWithContext(ctx, () => log(...))` → `await flush()` → captured body `[1]` is the order-record request. Capturing both inside a single ordered `it` makes the marker-then-record sequencing explicit and self-contained, removes the cross-test coupling that separate `it` blocks would hide (if the order-record test ran in isolation, `init`'s marker and the order record would drain into one two-record POST and fail `toEqual`), and best serves the "self-contained reference artifact" goal. Do **not** enable test shuffle. The captured-bodies array is module-scoped and cleared in `afterEach`; within the single ordered `it` it accumulates `[0]`=marker, `[1]`=record.
- **No normalization rule:** assert with strict `toEqual` against the parsed fixture JSON — never `toMatchObject`/subset/field-stripping. Strictness is what makes a deliberate field-shape regression (camelCase, integer `severityNumber`, hex ids, decimal-string `timeUnixNano`, `AnyValue` shapes) fail, satisfying the spec's regression requirement. Array `toEqual` is order-sensitive, so the attribute order `[level, order.id]` produced by `log()` (level first, then user attrs in `Object.keys` order) is verified too.

- [x] **Task 1: Scaffold the conformance test file and deterministic setup**
  Files: `test/conformance.node.test.ts`
  Create the new test file. Add a top-of-file `import '../src/node/index.js';` to register the Node `AsyncLocalStorage` context manager as a side effect (mirror the comment style in `test/winston.node.test.ts`). Import `init`, `log`, `flush` from `../src/core/index.js`; import `extract`, `objectCarrier`, `runWithContext` from `../src/node/index.js`. Import the three fixtures as JSON: `../contract/golden-record.json`, `../contract/fixtures/service-start.json`, `../contract/levels.json` (these resolve fine despite `tsconfig` `exclude: ["contract"]` — `resolveJsonModule` is on; `src/core/levels.ts` already imports `../../contract/levels.json`).
  In `beforeEach`:
  - `vi.useFakeTimers()`, `vi.setSystemTime(1718632800000)`
  - `vi.stubGlobal('crypto', { randomUUID: () => '0b9d7a3e-5f2c-4c1a-9e7d-3a6b8c1f2e4d' })` ⚠️ (see Critical design decision — **not** `vi.spyOn`)
  - install the `fetch` mock: `vi.stubGlobal('fetch', vi.fn(async (_url, opts) => { capturedBodies.push(opts.body); return new Response(null, { status: 200 }); }))` pushing each request body into a module-scoped `capturedBodies: string[]`.
  In `afterEach`: `vi.useRealTimers()`, `vi.unstubAllGlobals()`, `vi.restoreAllMocks()`, and reset `capturedBodies` (e.g. `capturedBodies.length = 0`).
  Add a small helper to parse the Nth captured body to an object (e.g. `const parsed = (i: number) => JSON.parse(capturedBodies[i] as string)`). Always `await flush()` before asserting so the mocked `fetch` resolves; rely on explicit `flush()` rather than advancing fake timers.

- [x] **Task 2: Assert the `service.start` marker against the fixture** (in the single ordered `it`; depends on Task 1)
  Files: `test/conformance.node.test.ts`
  In a `describe('OTLP wire conformance')` with one ordered `it('serializes service.start and a log record byte-for-byte', …)`: call `init({ project: 'example-project', service: 'example-service', endpoint: 'http://localhost:3100/otlp/v1/logs' })`, then `await flush()`. Assert `expect(JSON.parse(capturedBodies[0])).toEqual(serviceStartFixture)`. This validates the full envelope: resource attributes (with the stubbed instance id), `scope` `{ name: 'observe', version: '0.1.0' }`, `eventName: 'service.start'`, `body.stringValue: 'service.start'`, the `level`+`event.name` attributes, `severityNumber: 9`, `severityText: 'INFO'`, and the decimal-string timestamps.

- [x] **Task 3: Assert the canonical log record against the golden fixture** (same ordered `it`, immediately after Task 2; depends on Task 2)
  Files: `test/conformance.node.test.ts`
  Continuing in the **same** `it` (do not re-init — it is a no-op): build `const ctx = extract(objectCarrier({ traceparent: '00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01' }))`, assert `ctx` is defined, then `runWithContext(ctx, () => log('info', 'order placed', { 'order.id': 'A-1024' }))` and `await flush()`. Assert `expect(JSON.parse(capturedBodies[1])).toEqual(goldenRecordFixture)` — index `[1]` because the marker was captured at `[0]` earlier in the same test. This validates `traceId`/`spanId` lowercase-hex passthrough, `flags: 1`, the `level: 'info'` + `order.id: 'A-1024'` attributes in order, `body.stringValue: 'order placed'`, and `severityNumber: 9`/`severityText: 'INFO'`. Keeping marker capture and record capture in one ordered test guarantees the marker-before-record sequencing and avoids the isolation hazard described in the design decisions.

- [x] **Task 4: Assert the level table and pin the contract to v0.1.2** (depends on Task 1)
  Files: `test/conformance.node.test.ts`
  In a separate `describe('level table conformance')`, import `LEVELS` (and/or `severityFor`) from `../src/core/index.js` and assert **field-for-field** that for every token in `contract/levels.json` the SDK table returns the exact `severityNumber` (integer) and `severityText`, and that the key sets are equal (no missing, no extra) — use `it.each` over `Object.entries(levelsData.levels)`. Add a **contract-pin guard**: `expect(levelsData.version).toBe('0.1.2')` so a submodule at a different tag fails loudly (satisfies the spec's "fail loudly if at another tag"). Note (from review): this is a soft pin on the JSON `version` field, not a git-tag check — acceptable for this milestone (submodule is currently at `v0.1.2` and `levels.json` says `"0.1.2"`); it would not catch a detached commit still carrying `version: "0.1.2"`. This block intentionally overlaps `test/levels.test.ts` to keep the conformance suite a single self-contained reference artifact for swift/dart (where `levels.json` is a genuinely external oracle); that overlap is acceptable. It is somewhat tautological on the JS side because `src/core/levels.ts` derives `LEVELS` directly from the JSON — it earns its keep via the key-set-equality and version-pin guards, not by catching JS mapping bugs.
